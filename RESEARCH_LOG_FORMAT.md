# RESEARCH_LOG_FORMAT.md

เอกสารนี้กำหนดรูปแบบบันทึกการวิจัยชุดใหม่ของโครงการ **SPY 0DTE - Higanbana** บันทึกต้องเป็นงานวิจัยที่มนุษย์อ่านแล้วเข้าใจเหตุผลของการทดลอง ไม่ใช่การคัดลอกผลจาก script หรือ JSON มาเรียงต่อกัน

บันทึกเดิม `001` ถึง `041` ถูกย้ายไปเก็บใน `research_log/legacy_format/` โดยไม่แก้เนื้อหา ส่วนบันทึกชุดใหม่เก็บตรง `research_log/` และเริ่มนับจาก `001` ใหม่

## 1. หลักการเขียน

- เขียนเป็นภาษาไทยธรรมดา ใช้ภาษาอังกฤษเฉพาะชื่อเฉพาะ คำสั่ง path และศัพท์เทคนิคที่แปลแล้วทำให้ความหมายคลาดเคลื่อน
- ผู้อ่านต้องเข้าใจคำถาม วิธีทดลอง ผล และข้อจำกัดได้จากบันทึกเพียงไฟล์เดียว
- ตัวเลขสำคัญต้องมีคำอธิบายว่าหมายถึงอะไร ห้ามวางตารางแล้วปล่อยให้ผู้อ่านตีความเอง
- ต้องระบุระดับหลักฐาน เช่น `E0`, `E1`, `E2` และป้าย `under-sampled` หรือ `underpowered` ตามจริง
- ห้ามสรุปเกินหลักฐาน หากยังตอบคำถามไม่ได้ให้ใช้ข้อสรุป `ยังสรุปไม่ได้` พร้อมระบุข้อมูลที่ยังขาด
- ห้ามมี mojibake, replacement character หรือข้อความไทยที่เสียจาก encoding

## 2. การตั้งคำถามวิจัย

คำถามวิจัยเป็นจุดเริ่มต้นของทั้งบันทึก ต้องสั้น ชัด และมีขอบเขต ไม่ใช่คำถามลอย ๆ

คำถามที่ดีต้องระบุให้เห็นอย่างน้อยว่า:

1. กำลังศึกษาอะไรหรือเปรียบเทียบอะไร
2. ใช้กับระบบ กลยุทธ์ หรือกลุ่มข้อมูลใด
3. อยู่ภายใต้เงื่อนไขหรือช่วงเวลาใด
4. ใช้ผลลัพธ์ใดเป็นตัวตัดสิน

ควรมีคำถามหลักเพียงหนึ่งข้อ ความยาวหนึ่งถึงสองประโยค และต้องตอบได้ด้วยข้อมูลจากการทดลองนี้

ตัวอย่างที่ไม่ควรใช้:

- “ORB ใช้ได้ไหม” เพราะไม่ระบุช่วงตลาด กฎ หรือเกณฑ์ตัดสิน
- “LLM ช่วยได้หรือไม่” เพราะไม่ระบุหน้าที่ของ LLM ข้อมูลที่ให้ หรือผลลัพธ์ที่ต้องเปรียบเทียบ
- “ระบบดีขึ้นหรือเปล่า” เพราะไม่ระบุว่าดีขึ้นด้านใดและเทียบกับอะไร

ตัวอย่างที่ชัดกว่า:

- “ภายใต้ prior-close VIX ต่ำกว่า 25 และไม่มีข่าวมหภาคสำคัญ กฎ ORB เวลา 09:35 ทำให้ SPY 0DTE call/put debit spread หลังต้นทุนมี Expected PnL เป็นบวกในข้อมูล OOS ที่ไม่เคยใช้ปรับกฎหรือไม่”
- “คะแนนข่าวแบบ market-based ก่อน 10:00 ET ลด ES99 ของ Sub-System A ได้อย่างน้อย 15% เมื่อเทียบกับ quantitative-only และ semantic filter หรือไม่”

ทุกบันทึกใหม่ต้องมีข้อความสามบรรทัดนี้ในส่วนที่ 2:

```text
คำถามวิจัย: <คำถามหลักหนึ่งข้อ>
ขอบเขต: <ระบบ ข้อมูล ช่วงเวลา เงื่อนไข และสิ่งที่ไม่รวม>
สมมติฐาน: <ผลที่คาดว่าจะพบและเหตุผลเชิงกลไก>
```

สมมติฐานต้องตอบคำถามโดยตรง ระบุทิศทางของผลที่คาด และสามารถถูกหักล้างได้ ห้ามเขียนเพียงว่า “น่าจะดีขึ้น” หรืออธิบายเหตุผลโดยไม่มีผลที่วัดได้

## 3. ชื่อไฟล์และเวลา

ใช้ชื่อไฟล์:

```text
NNN-higanbana-<clear-topic-slug>.md
```

- `NNN` เป็นเลขสามหลักต่อเนื่องของชุดใหม่ บันทึกถัดไปคือ `001`
- slug ต้องสื่อคำถามหรือหัวข้อจริง ห้ามใช้ชื่อกว้าง เช่น `test`, `new-log`, `experiment`
- หากแก้บันทึกเดิม ให้ใช้เลขเดิม หากเป็นการทดลองรอบใหม่ ให้ใช้เลขถัดไป
- Timestamp ใช้ UTC รูปแบบ `YYYY-MM-DDThh:mm:ssZ`

## 5. กฎการอภิปรายและสรุปผล

- แยก “ผลลัพธ์” ออกจาก “อภิปรายผล” ให้ชัด ส่วนที่ 4 บอกสิ่งที่วัดได้ ส่วนที่ 5 อธิบายว่าผลนั้นหมายถึงอะไรและอาจผิดได้อย่างไร
- หากมีการค้นหาหลาย trial ต้องระบุ search log และ DSR หรือ blocker ที่ทำให้ยังคำนวณ DSR ไม่ได้
- หากรายงาน Sharpe ต้องรายงาน N, MinTRL, PSR และป้าย sample adequacy ควบคู่กัน
- หากใช้ข้อมูลออปชัน ต้องแยก `mid PnL` กับ `implementable PnL` และเปิดเผยต้นทุน
- หากเป็น proxy ต้องเขียนชัดว่า proxy ตอบอะไรได้และแทน exact evidence ไม่ได้ตรงไหน
- หากผลขึ้นกับ filter ต้องรายงานจำนวนตัวอย่างที่เหลือหลังแต่ละ filter
- สรุปข้อถัดไปต้องแก้ข้อจำกัดของรอบนี้ ไม่ใช่เพิ่มการทดลองใหม่โดยไม่มีเหตุผล

## 6. เมื่อใดต้องเขียนบันทึก

ต้องเขียนหรือปรับบันทึกเมื่อ:

- จบการทดลองหรือ diagnostic ที่ให้ข้อสรุปเชิงวิจัย
- หักล้าง park หรือจำกัดขอบเขตของสมมติฐานจากหลักฐานใหม่
- พบว่าคำถามหรือสมมติฐานเดิมตั้งผิดและมีการตั้งคำถามใหม่อย่างเป็นทางการ
- แก้ผลการทดลองเดิมเพราะพบข้อผิดพลาดที่กระทบข้อสรุป

ไม่ต้องสร้างบันทึกใหม่สำหรับ preregistration, cost plan, data acquisition หรือการแก้ระบบที่ยังไม่มีผลเชิงวิจัย เว้นแต่การกระทำนั้นเองตอบคำถามวิจัยที่ลงทะเบียนไว้

1. ตรวจว่าชื่อไฟล์และหมายเลขต่อเนื่อง

4. อ่านคำถามอีกครั้งว่าคำถามสั้น ชัด มีขอบเขต และตอบได้ด้วยข้อมูลรอบนี้
5. ตรวจว่าไม่มี mojibake และไม่มีข้อสรุปเกิน evidence tier
6. รัน:
