# RESEARCH_LOG_FORMAT.md

เอกสารนี้เป็นกติกาสำหรับบันทึกงานวิจัยของโปรเจกต์ **SPY 0DTE - Higanbana** ทุกไฟล์ใน `research_log/` ต้องอ่านรู้เรื่องเหมือนบันทึกที่นักวิจัยเขียนให้มนุษย์อ่าน ไม่ใช่การ dump ผลจาก script

## 1. หลักภาษา

- เขียนเป็นภาษาไทยธรรมดา อ่านแล้วเข้าใจได้จาก GitHub โดยไม่ต้องเปิด artifact ประกอบทันที
- ใช้ภาษาอังกฤษเฉพาะคำศัพท์เทคนิคที่ควรคงไว้ เช่น `backtest`, `proxy`, `implementable PnL`, `MinTRL`, `PSR`, `DSR`, `E1`, `OOS`, `paper trading`
- ถ้าจำเป็นต้องใช้ศัพท์อังกฤษ ให้ประโยคนั้นยังเป็นภาษาไทย ไม่เขียนสลับจนอ่านเหมือนภาษาต่างดาว
- ห้ามเขียนเหมือน log เครื่องหรือ JSON summary ต้องเล่าเป็นลำดับ: คำถาม -> วิธีตรวจ -> ผลที่พบ -> ความหมาย -> ข้อจำกัด -> ก้าวต่อไป
- ห้ามมี mojibake หรือข้อความเพี้ยน เช่น `mojibake marker`, `replacement character`, หรือข้อความไทยที่กลายเป็นอักขระอ่านไม่ออก
- ถ้าหลักฐานยังไม่พอ ต้องเขียนตรง ๆ ว่า `ยังสรุปไม่ได้` และอธิบายว่าขาดอะไร ห้ามทำให้ผลดูดีเกินหลักฐานจริง

## 2. การตั้งชื่อไฟล์

ใช้รูปแบบ:

```text
NNN-higanbana-<clear-topic-slug>.md
```

กฎ:

- `NNN` เป็นเลขสามหลักต่อเนื่อง เช่น `001`, `002`, `036`
- `higanbana` เป็นคำคงที่
- `<clear-topic-slug>` ต้องอ่านแล้วเข้าใจหัวข้อ เช่น `h-a2-normal-control-exact-replay`
- ห้ามใช้ชื่อคลุมเครือ เช่น `test`, `new-log`, `exp07`
- ถ้าแก้บันทึกของการทดลองเดิม ให้แก้ไฟล์เลขเดิม ไม่สร้างเลขใหม่
- ถ้าเป็นการทดลองรอบใหม่จริง ให้ใช้เลขถัดไป

## 3. Timestamp

- ใช้ UTC เท่านั้น
- รูปแบบ: `YYYY-MM-DDThh:mm:ssZ`
- ตัวอย่าง: `2026-07-07T10:30:00Z`

## 4. โครงสร้างบังคับ 6 ส่วน

## 5. กฎการเขียนผล

- รายงานทั้งผลที่ดีและผลที่ไม่ผ่าน
- ถ้ามีตัวเลขมากกว่า 2-3 ค่า ให้ใช้ตาราง แต่ต้องมีประโยคอธิบายความหมายหลังตาราง
- ต้องระบุ `evidence tier` และข้อจำกัดของหลักฐาน
- ถ้า sample เล็ก ให้ระบุ `under-sampled` หรือ `underpowered` ตามจริง
- ถ้ามีการเลือกค่าที่ดีที่สุดจากหลาย trial ต้องพูดถึง search log และ DSR/DSR blocker
- ถ้าการทดลองยังเป็น `proxy` ต้องเขียนชัดว่าไม่ใช่ exact evidence
- ถ้าไม่ได้วัด metric ใด ห้ามเดา ให้เขียนว่าไม่ได้วัด
- อย่าเขียนว่า “pipeline ผ่าน” แล้วจบ ต้องอธิบายว่าผลนั้นช่วยตอบสมมติฐานอะไร และยังตอบอะไรไม่ได้

## 6. เมื่อใดต้องเขียน Research Log

ต้องเขียนหรือแก้ research log เมื่อ:

- จบการทดลองหนึ่งรอบ
- จบ diagnostic ที่ให้ข้อสรุปเชิงวิจัย
- ยกเลิกหรือ park hypothesis เพราะหลักฐานไม่พอ
- แก้ผลการทดลองเดิมให้ถูกต้องขึ้น

ไม่จำเป็นต้องเขียน log ใหม่เมื่อ:

- สร้าง preregistration เฉย ๆ
- แก้ validator/test/audit โดยยังไม่มีผลวิจัยใหม่
- ทำ data acquisition ที่ยังเป็น E0 control artifact และยังไม่เกิด diagnostic result
